uv

uv 是一个用 Rust 编写的 Python 工具，目标是把“Python 项目从创建到运行”的整个链路做成一个又快又简单的统一工具。它之所以要存在，本质原因只有一个：Python 的包管理和环境管理长期处于碎片化、低效和高复杂度的状态。

在 uv 出现之前，一个稍微正规的 Python 项目，通常需要一堆工具配合使用。比如用 pyenv 管理 Python 版本，用 venv 或 virtualenv 管理虚拟环境，用 pip 安装依赖，用 pip-tools 或 poetry-lock 锁版本，用 poetry 或 setuptools 管理项目元数据。这些工具各自都能用，但组合在一起之后，慢、复杂、容易出错，而且在 CI、跨平台、团队协作中经常踩坑。尤其是 pip 的性能问题，在依赖多一点时会非常明显，解析依赖和下载包都很慢。

uv 的设计目标就是：把这些事情合并，用一个工具解决，并且在性能上做到数量级的提升。它不是在 pip 上包一层，而是从底层重新实现了解析、下载和缓存机制，所以速度非常快，官方和社区的实测通常是 pip 的 10 到 100 倍。

从功能上看，uv 至少做了几件关键的事情。第一，它是一个极速的包安装器，可以直接替代 pip 使用。你可以用 uv pip install 来安装依赖，语义和 pip 几乎一致，但速度快很多，而且缓存策略更激进、更合理。第二，它自带虚拟环境管理能力，可以创建、使用和复用虚拟环境，不需要再单独依赖 venv 或 virtualenv。第三，它支持项目级的依赖管理，能够读取 pyproject.toml，并生成和使用锁文件，从而保证依赖的可复现性，这一点在 CI 和生产环境里非常重要。第四，它内置 Python 解释器的发现和管理逻辑，能自动选择合适的 Python 版本，减少“系统 Python / 项目 Python / 虚拟环境 Python”之间的混乱。

更深一层看，uv 的价值不只是“更快的 pip”，而是它在试图把 Python 的开发体验向现代语言工具链对齐。像 Rust 有 cargo，Node.js 有 npm 和 pnpm，Go 有 go tool，而 Python 长期缺少一个同时覆盖版本、环境、依赖和项目生命周期的统一入口。uv 正是在填这个空位，而且它的实现方式是工程化、性能优先、偏底层的，这也是它选择用 Rust 而不是 Python 实现的原因。

总结来说，uv 存在的原因是 Python 工具链太慢、太散、太容易出问题；它的核心功能是极速依赖安装、虚拟环境管理、项目依赖锁定和 Python 版本协同；它解决的问题不是“多一个选择”，而是试图成为那个你可以默认使用、不用纠结的基础工具。如果你关心可复现性、CI 速度、开发体验和长期维护成本，uv 基本上是一个顺理成章的选择。

安装

Macos and Linux

curl -LsSf https://astral.sh/uv/install.sh | sh

Windows

powershell -ExecutionPolicy ByPass -c "irm https://astral.sh/uv/install.ps1 | iex"

如果碰到错误，可以去官网看看：https://docs.astral.sh/uv/getting-started/installation/#standalone-installer

基本用法

创建虚拟环境：

uv venv

只要进入一个文件夹，输入该指令，当前文件夹名称会变成环境名，之后用source .venv/bin/activate 激活虚拟环境。

激活之后就可以用了。

安装包

uv pip install xxx

它也可以加镜像。
